# Portfolio summary

`internal-assistant-mvp` es un asistente conversacional interno pensado para una empresa ficticia de logistica. La idea central es simple: una persona del equipo hace una pregunta sobre operaciones, seguridad, onboarding o politicas internas, y el asistente responde usando solo informacion recuperada desde un indice RAG construido sobre documentacion privada y tickets historicos. Si el sistema no tiene evidencia suficiente, no inventa el procedimiento: pide aclaracion y, si el problema sigue sin resolverse, ofrece registrar una incidencia nueva.

La eleccion de PostgreSQL + pgvector no es casual. Para un MVP de portfolio queria un stack que resolviera varias cosas a la vez: persistencia transaccional, full-text search, soporte de vectores, coste contenido y una historia de despliegue razonable tanto en local como en Azure. Con PostgreSQL puedo mantener en una sola base incidents, documents, chunks, conversations, feedback y retrieval logs. Eso simplifica la demo y hace mas visible una decision tecnica importante: no hace falta introducir un buscador separado desde el primer dia para demostrar un flujo RAG serio. Para este nivel de alcance, pgvector y el buscador textual nativo de PostgreSQL cubren lo esencial y permiten explicar con claridad el tradeoff frente a un servicio gestionado como Azure AI Search.

La arquitectura esta separada en tres piezas principales. `app-service` expone la API conversacional y el endpoint compatible con Bot Framework / Teams. `custom-incidents-api-function` simula el sistema fuente de incidencias: el bot no escribe tickets directamente en la tabla final, sino que los crea pasando por una API fuente. `indexer-function` es la pieza que chunkifica e indexa contenido en PostgreSQL. Esa separacion es deliberada y demuestra una idea importante para sistemas empresariales: el indice RAG no tiene por que ser la fuente de verdad. Se puede desacoplar la persistencia operativa del material optimizado para retrieval.

El proyecto intenta evitar alucinaciones de varias formas. La primera es estructural: el prompt del modelo se alimenta con contexto recuperado y el asistente expone fuentes y fragmentos. La segunda es de flujo: si el retrieval devuelve poca confianza o no trae evidencia suficiente, el sistema fuerza una aclaracion en lugar de continuar respondiendo con seguridad falsa. La tercera es operativa: tras dos aclaraciones fallidas, el flujo cambia de objetivo y propone registrar una incidencia no resuelta. Esa transicion importa mucho en una demo, porque muestra que el asistente no se limita a contestar "no se", sino que encadena una accion util para el negocio.

Otro aspecto que queria demostrar es la trazabilidad. Cada respuesta puede devolver fuentes, el sistema guarda retrieval logs, y el proyecto incluye una capa de evaluacion RAG con datasets curados y reportes JSON/Markdown. Eso convierte el asistente en algo medible. No solo puedo decir que responde "bien" o "mal"; puedo medir hit@k, MRR, coverage de fuentes, precision de abstencion y comportamiento ante casos adversariales. Para portfolio esto es especialmente valioso, porque deja ver una preocupacion de ingenieria mas madura: un asistente no se queda terminado cuando responde por consola, sino cuando puedes observarlo, compararlo y argumentar por que una configuracion es mejor que otra.

Tambien esta pensado para una demo barata. En local, la ruta principal usa Docker Compose para PostgreSQL, scripts reproducibles para seed e indexado, y providers `mock`, `openai` u `openai_compatible`. En cloud, la historia se apoya en Azure App Service, Azure Functions, PostgreSQL Flexible Server y Azure Bot, pero con SKUs pequenos y scripts para parar PostgreSQL cuando no se usa. Esto deja una narrativa muy util para portfolio: el mismo proyecto es entendible y usable por un tercero que quiera levantarlo en local, pero tambien es ensenable en un entorno cloud real sin disparar costes ni introducir demasiada complejidad enterprise.

El proyecto no esta planteado como producto final. No hay SSO, no hay RBAC real, no hay hardening de secretos con Key Vault como requisito, no hay reranking avanzado ni politicas complejas por departamento. Eso es intencional. La prioridad aqui es mostrar criterio tecnico: una arquitectura clara, tradeoffs razonables, un flujo funcional creible y una ruta de evolucion bien explicada. En otras palabras, primero demostrar fundamento y capacidad de entrega; despues ya llegarian las capas enterprise.

Si este sistema evolucionara, el camino mas natural seria reforzar tres areas. La primera es seguridad y gobierno: mover secretos a Key Vault, introducir identidad corporativa y aplicar permisos por area o por sistema. La segunda es calidad RAG: mejorar datasets, retrieval tuning, robustez ante prompt injection y cobertura semantica. La tercera es experiencia de uso: cards de Teams mas ricas, mejores workflows conversacionales y tooling de observabilidad mas profundo. El valor del MVP es que deja ese camino abierto sin esconder las limitaciones actuales.

Como pieza de portfolio, el proyecto demuestra varias competencias al mismo tiempo: backend Python, modelado de datos, RAG, evaluacion, despliegue cloud y criterio de producto tecnico. Pero la parte mas importante es probablemente la mas sencilla de explicar: no es un chatbot generico. Es un asistente interno con limites claros, con una separacion sana entre fuente de verdad e indice, y con una forma reproducible de medir si realmente esta recuperando y respondiendo mejor.
